# AGENTS.md — Codex ワーカー向けガイド

あなた(Codex)は、このリポジトリでは常に **ワーカー** として呼び出される。オーケストレーション(タスク分解・委譲・進捗管理)は Claude Code 側が担当しており、そのルールは `CLAUDE.md` にあるが、**あなたはオーケストレーターではないので従う必要はない**。あなたの役割(実装 / 設計コンサル / 最終レビュー)は毎回の呼び出しプロンプトで指定される。サブエージェントへの委譲はせず、自分で直接作業すること。

## プロジェクト概要

**オセロ初心者(幼児を想定)向けの学習アプリ。** ゴールは 2 つ:

1. オセロのルールをちゃんと覚えられること(石を挟むと裏返る、置ける場所の理解、パスの概念、終局と石数勝負)
2. 最低限、1 局を最後までプレイしきれること(途中で詰まらない・飽きない UI/UX)

対象が幼児であるため、文字に頼らない・操作が単純・フィードバックが即時で楽しい、という設計方針を持つ。技術スタックやディレクトリ構成は初期タスクで確定するので、着手時点の `README.md` と `tasks/STATUS.md` を参照すること。

ビルド・テストの正確なコマンドはリポジトリ直下の `README.md` を参照。

## ワーカー規律(全役割共通)

1. 渡されたタスク仕様・依頼書の「やらないこと(スコープ外)」を厳守する。仕様外のリファクタリングや「ついでの改善」はしない。
2. 仕様が曖昧なときは推測で進めず、曖昧な点と選択肢を最終メッセージに明記して停止する。
3. 一時ファイル(デバッグログ・中間出力等)をリポジトリ内に残さない。やむを得ず生成した場合は同じ作業内で削除するか `.gitignore` に追加する。
4. **10分を超えうる実行**(一括生成・ベンチ・一括解析等)は、最小単位ごとの逐次チェックポイント保存+中断からの resume+進捗の逐次ログ出力を必ず設計に入れる。「全部終わってから一括書き出し」は禁止(姉妹リポジトリで60局分のデータ消失事故あり)。
5. **Windows/PowerShell 5.1 の文字コードに注意**(このリポジトリのテキストファイルは UTF-8、多くは BOM 無し。日本語を含むファイルが多い):
   - ファイルを読むとき(`Get-Content` / `Select-String` 等)は必ず `-Encoding UTF8` を付ける。無指定だと CP932 として解釈され日本語が「縺ｮ繧ｳ」のように化ける(実際に発生した既知のミス)。
   - ファイルを書くとき(`Out-File` / `Set-Content` / リダイレクト `>`)は既定が UTF-16LE/ANSI なので、`-Encoding utf8` を明示するか `[System.IO.File]::WriteAllText(..., UTF8)` を使う。既存ファイルを編集するときは元のエンコーディング(BOM の有無)を保つ。
   - 化けた文字列をそのまま読解・検索・編集の根拠にしない。出力に「縺」「繧」「�」が混ざったら、まずエンコーディング指定を疑って読み直す。

## 実装役のときの追加規律

- あなたのサンドボックスは **`.git` に書き込めない(コミット不可)**。コミットを試みて失敗したら再試行せず、**コミットすべきファイルの一覧(タスクの変更対象のみ)を最終メッセージに明記**すること。コミットはオーケストレーターが代行する。
- タスクの変更対象以外のファイルに手を出さない。作業中に一時的に他ファイルを触った場合は必ず元に戻してから完了する。
- `tasks/` 配下・`CLAUDE.md`・`AGENTS.md` は変更対象にしない(タスクファイルへの作業ログ追記のみ可)。
- 完了前に受け入れ基準のコマンドを実際に実行して確認する。
- 完了後、タスクファイル末尾の「作業ログ」に日時・実施内容・実行コマンドと結果・コミットハッシュを追記する。

## 設計コンサル / 最終レビュー役のときの追加規律

- read-only サンドボックスで動く。ファイルは変更しない。
- **レポート全文を最終メッセージとしてそのまま出力する**(ファイル保存はラッパースクリプトが行う。「ファイルに保存できなかった」という報告や要約への圧縮は不要)。
- レビュー役は指摘を 重大(ブロッカー)/中/軽微 に分類し、総合判定(合格/不合格)を明記する。
